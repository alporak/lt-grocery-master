## STEP 1: Create Cloudflare Account
- Go to https://dash.cloudflare.com/sign-up
- Create an account (email + password). Free plan is sufficient.
- Verify email if prompted.

**Verify:** User confirms "Cloudflare account created."

---

## STEP 2: Add krepza.lt Domain to Cloudflare
- In Cloudflare dashboard, click **"Add a Site"** (or **"+ Add a domain"**)
- Enter: `krepza.lt`
- Select the **Free** plan (bottom of page — $0/month)
- Cloudflare will scan existing DNS records. Continue past this — DNS will be managed automatically when the tunnel is created.
- Cloudflare will display TWO nameservers (e.g., `alice.ns.cloudflare.com` and `bob.ns.cloudflare.com`). **Copy both nameserver addresses.**

**Verify:** Cloudflare dashboard shows krepza.lt with status "Pending Nameserver Update."

---

## STEP 3: Update Nameservers at Domain Registrar
- Log into the registrar where krepza.lt was purchased (e.g., hostinger.lt, iv.lt, serveriai.lt, etc.)
- Find the domain's **Nameserver / DNS settings** page.
- Replace existing nameservers with the two Cloudflare nameservers from Step 2.
- Save changes.

⚠️ **DNS propagation takes 24-48 hours.** The site will not be immediately accessible. This is normal — do not undo the change.

**Verify (can be checked later):** Run `dig krepza.lt NS` from the VPS. After propagation, output will show Cloudflare nameservers:
```
krepza.lt.  86400  IN  NS  alice.ns.cloudflare.com.
krepza.lt.  86400  IN  NS  bob.ns.cloudflare.com.
```
If the output still shows the old registrar's nameservers, DNS has not propagated yet. Wait and re-check.

---

## STEP 4: Set SSL/TLS to Full (strict)
- In Cloudflare dashboard, go to **SSL/TLS → Overview**
- Change encryption mode from "Flexible" to **"Full (strict)"**
- This ensures Cloudflare validates the origin certificate (tunnel provides origin cert automatically).

**Verify:** SSL/TLS overview page shows "Full (strict)" as the active mode.

---

## STEP 5: Create Cloudflare Tunnel
- In Cloudflare dashboard, go to **Zero Trust** (left sidebar, may need to click "More" to see it).
  - If Zero Trust is not visible: navigate to https://one.dash.cloudflare.com/
- In Zero Trust, go to **Networks → Tunnels**
- Click **"Create a tunnel"**
- Name the tunnel: `krepza-tunnel` (or any name)
- Click **"Save tunnel"**
- On the next screen, you'll see installation options. Select **"Docker"** as the environment.
- Cloudflare displays a command with `--token <long-base64-string>`. **Copy only the token value** (the long base64 string — NOT the full docker run command).

The token looks like: `eyJh...very-long-base64-string...`

**DO NOT run the docker command shown by Cloudflare.** The project uses docker-compose (configured in Plan 08-01), not `docker run`.

---

## STEP 6: Place TUNNEL_TOKEN in .env
- On the VPS, edit the `.env` file (same directory as docker-compose.yml):
  ```bash
  # Add the token you copied from Cloudflare:
  TUNNEL_TOKEN=eyJh...your-actual-token...
  ```
- If `.env` doesn't exist, copy from `.env.example`: `cp .env.example .env` then fill in all required values (AUTH_SECRET, OAuth keys, etc.) PLUS the TUNNEL_TOKEN.

⚠️ **CRITICAL:** Verify `.env` is gitignored:
```bash
grep '\.env' .gitignore
```
Should show `.env` is listed. The token MUST NOT enter git history.

---

## STEP 7: Configure Cloudflare Page Rule for www → apex redirect (per D-01)
- In Cloudflare dashboard, go to **Rules → Page Rules** (or **Rules → Redirect Rules** if using the newer UI)
- Create a rule:
  - URL pattern: `www.krepza.lt/*`
  - Setting: **Forwarding URL** (301 Permanent Redirect)
  - Destination: `https://krepza.lt/$1`
- This ensures `www.krepza.lt` redirects to `krepza.lt`.

---

## STEP 8: Configure Tunnel Ingress Rules (per D-01)
- In Cloudflare Zero Trust → Networks → Tunnels → click on `krepza-tunnel` → **Configure**
- Go to the **Public Hostname** tab (or "Ingress Rules")
- Add a public hostname:
  - **Subdomain:** (leave empty for apex) OR enter `@` if required
  - **Domain:** `krepza.lt`
  - **Service Type:** HTTP
  - **URL:** `web:3131`
- Add a SECOND public hostname for www:
  - **Subdomain:** `www`
  - **Domain:** `krepza.lt`
  - **Service Type:** HTTP
  - **URL:** `web:3131`
- Save the tunnel configuration.

**Verify:** Tunnel status shows "HEALTHY" in the dashboard (may take a minute after cloudflared container starts — the container isn't running yet, that happens in Plan 08-03).

---

## Google OAuth (Google Cloud Console)

1. Go to https://console.cloud.google.com/apis/credentials
2. Select the project used for Krepza OAuth
3. Under **OAuth 2.0 Client IDs**, click the client ID for the web application
4. In the **Authorized redirect URIs** section, click **"+ ADD URI"**
5. Enter EXACTLY: `https://krepza.lt/api/auth/callback/google`
6. If a localhost URI exists (`http://localhost:3131/api/auth/callback/google`), keep it for development — do NOT remove it (unless you never develop locally).
7. Click **Save**

**Verify:** The redirect URI list includes `https://krepza.lt/api/auth/callback/google`

---

## Facebook OAuth (Facebook Developers)

1. Go to https://developers.facebook.com/apps
2. Select the Krepza app
3. In the left sidebar, go to **Facebook Login → Settings** (under "Products")
4. Under **Valid OAuth Redirect URIs**, enter EXACTLY: `https://krepza.lt/api/auth/callback/facebook`
5. If a localhost URI exists (`http://localhost:3131/api/auth/callback/facebook`), keep it for development — do NOT remove it.
6. Click **Save Changes**

**Verify:** The Valid OAuth Redirect URIs list includes `https://krepza.lt/api/auth/callback/facebook`

---

⚠️ **IMPORTANT — NEXTAUTH_URL dependency:** 
The next-auth library in packages/web/src/lib/auth.ts uses `NEXTAUTH_URL` to construct callback URLs automatically. No code changes are needed because:
- GoogleProvider and FacebookProvider use `process.env.GOOGLE_CLIENT_ID!` and `process.env.GOOGLE_CLIENT_SECRET!` — they derive callback URLs from NEXTAUTH_URL.
- The OAuth provider dashboards only need to ALLOW the callback URL — they don't generate it.
- Setting `NEXTAUTH_URL=https://krepza.lt` (done in Plan 08-01's .env.example, and in .env on the VPS) ensures next-auth generates `https://krepza.lt/api/auth/callback/google` and `https://krepza.lt/api/auth/callback/facebook`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User browser → Cloudflare dashboard | User authenticates with Cloudflare credentials to create tunnel and configure domain. |
| User browser → Google Cloud Console / Facebook Developers | User authenticates with provider credentials to update OAuth redirect URIs. |
| TUNNEL_TOKEN → .env file | Token moves from Cloudflare dashboard to VPS filesystem. Must not traverse git or any public channel. |
| OAuth callback → krepza.lt | Production callback URLs use https://krepza.lt. Mismatch with NEXTAUTH_URL causes login failure. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-06 | Information Disclosure | TUNNEL_TOKEN transit from dashboard to .env | mitigate | User copies token directly from Cloudflare dashboard to .env on VPS. No intermediaries. Agent does not see or log the token. Token is gitignored. |
| T-08-07 | Spoofing | Cloudflare account compromise | accept | Cloudflare account security is user's responsibility. Use strong password + 2FA on Cloudflare account. Out of scope for this phase. |
| T-08-08 | Information Disclosure | OAuth client secrets exposed in console screenshots | mitigate | Agent instructs user: do not share screenshots of OAuth console pages containing client secrets. |
| T-08-09 | Repudiation | OAuth callback URL mismatch (http vs https) | mitigate | Plan 08-01 sets NEXTAUTH_URL=https://krepza.lt. This plan configures https:// callback URLs in both consoles. Verification in Plan 08-03 confirms login works. Pitfall #5 documented. |
| T-08-10 | Denial of Service | DNS propagation delay prevents tunnel from working | accept | DNS propagation is inherent to the system. Monitor with `dig krepza.lt NS` and wait. Not a security threat — operational concern. |
</threat_model>
