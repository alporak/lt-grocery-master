  <action>
Execute these verification steps in order. Stop at any failure and report the specific error.

**STEP 1: Start the cloudflared container**
```bash
docker compose up -d cloudflared
```
This starts ONLY the cloudflared service (web/scraper/embedder should already be running). The `-d` flag runs it detached.

**STEP 2: Check cloudflared logs for successful connection**
```bash
docker compose logs cloudflared --tail 30
```
Look for log lines indicating a successful Cloudflare connection:
- `INF Registered tunnel connection` or similar — confirms tunnel connected to Cloudflare
- `INF Connection <id> registered with protocol: quic` — QUIC protocol connection
- If you see `ERR` or connection failures: check TUNNEL_TOKEN in .env, verify the tunnel exists in Cloudflare dashboard.

**STEP 3: Verify internal Docker network routing**
```bash
docker compose exec cloudflared curl -s -o /dev/null -w "%{http_code}" http://web:3131/
```
Expected output: `200` (or `302` redirect, or any non-5xx).
If this returns `000` or `502`: cloudflared cannot reach the web container. Check that both services are on the default Docker network (`docker compose ps`).

Also verify the web container is responding:
```bash
docker compose exec cloudflared curl -s http://web:3131/ | head -c 200
```
Should return HTML content (the Krepza homepage) — NOT an error page.

**STEP 4: Check DNS propagation**
```bash
dig +short krepza.lt
```
If this returns Cloudflare IPs (e.g., `104.21.x.x` or `172.67.x.x`): DNS has propagated ✓
If this returns the old registrar IPs or nothing: DNS has NOT propagated yet. Note: this may take 24-48 hours.

Also verify nameservers:
```bash
dig +short krepza.lt NS
```
Should show Cloudflare nameservers (e.g., `alice.ns.cloudflare.com.`, `bob.ns.cloudflare.com.`).

**STEP 5: Test HTTPS via tunnel (only if DNS has propagated)**
```bash
curl -s -o /dev/null -w "%{http_code}" https://krepza.lt
```
Expected: `200`
If `000` or connection refused: DNS may not have propagated yet.
If `502 Bad Gateway`: Tunnel connects to Cloudflare but can't reach web:3131 — check Step 3.
If `525 SSL handshake failed`: Check Cloudflare SSL/TLS mode is "Full (strict)" in dashboard.

⚠️ If DNS has NOT propagated yet (Step 4 shows old IPs): log a message "DNS propagation pending — will retry verification in . The tunnel infrastructure is in place and ready. Run `dig krepza.lt` periodically to check propagation, then re-run Step 5."

**STEP 6: Verify HTTP→HTTPS redirect**
```bash
curl -sI -o /dev/null -w "%{http_code} %{redirect_url}" http://krepza.lt
```
Expected: `301 https://krepza.lt` (redirect from http to https). Cloudflare handles this automatically when SSL is enabled.

Now that the tunnel is verified and OAuth is working (Task 2 approved), remove the direct port exposure from the web service per D-02 incremental migration.

**Option A (recommended — keep localhost-only access for debugging):**
Replace line 6-7 in docker-compose.yml:
```yaml
    ports:
      - "3131:3131"
```
with:
```yaml
    ports:
      - "127.0.0.1:3131:3131"
```
This binds port 3131 ONLY to localhost (127.0.0.1), not to 0.0.0.0. External traffic cannot reach the port — only Cloudflare Tunnel via Docker internal network can. Local dev/debugging on the VPS still works via `curl http://localhost:3131`.

**Option B (fully remove port — cleaner):**
Remove the `ports:` block entirely (delete lines 6-7). The web container listens on port 3131 internally (within Docker network) but nothing is mapped to the host.

Choose the option that matches the user's environment:
- If the VPS is a dedicated server and you may need to debug locally → use Option A (127.0.0.1 binding)
- If you want maximum security and zero open ports → use Option B (remove entirely)

**After editing, restart the web container:**
```bash
docker compose up -d web
```

**Verify no public-facing ports:**
```bash
ss -tlnp | grep 3131
```
Expected: No output (port not listening) OR shows only `127.0.0.1:3131` (localhost only).

**Verify tunnel still works after port change:**

Expected: `200` — tunnel still routes traffic correctly even without host port exposure.

Expected: `200`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User browser → https://krepza.lt | Public HTTPS traffic. Cloudflare Edge provides SSL termination, DDoS, WAF. |
| Cloudflare Edge → cloudflared | Encrypted QUIC/HTTP2 tunnel authenticated by TUNNEL_TOKEN. |
| cloudflared → web:3131 | Internal Docker network (bridge). Unencrypted HTTP — acceptable because both containers are on the same host. |
| web:3131 → OAuth providers | Outbound HTTPS to Google/Facebook APIs. Standard TLS. |
| Host network → web:3131 (removed) | Trust boundary eliminated by removing port exposure in Task 3. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-11 | Elevation of Privilege | Direct access to web:3131 bypassing Cloudflare | mitigate | Port exposure removed (Task 3 — D-02). Either bound to 127.0.0.1 only or fully removed. Public traffic MUST traverse Cloudflare Tunnel. Verified with `ss -tlnp`. |
| T-08-12 | Tampering | MITM between cloudflared and web container | accept | Internal Docker bridge network on single host. Attacker would need root access. Acceptable for single-server self-hosted deployment. |
| T-08-13 | Spoofing | Fake OAuth callback (redirect_uri manipulation) | mitigate | OAuth providers validate redirect URI against their configured list. Both Google and Facebook consoles have `https://krepza.lt/api/auth/callback/*` registered. next-auth validates state parameter to prevent CSRF. |
| T-08-14 | Information Disclosure | Session cookies not Secure due to http NEXTAUTH_URL | mitigate | NEXTAUTH_URL=https://krepza.lt (verified in Plan 08-01). Browsers set `Secure` flag on cookies delivered over HTTPS. next-auth sets `secure: true` when NEXTAUTH_URL starts with https://. |
| T-08-15 | Denial of Service | Tunnel connection failure | accept | cloudflared auto-reconnects on disconnect. `restart: unless-stopped` recovers from container crash. Cloudflare edge absorbs DDoS. No redundant tunnel — acceptable for single-server. |
</threat_model>
