  <action>
Add `logging:` configuration to all 5 services (web, scraper, embedder, cloudflared, backup) in docker-compose.yml to enable structured JSON logging with size-based rotation. Per D-07 and HRDN-08: json-file driver, max-size 10m, max-file 3.

For EACH service, add this block after the `restart:` line (and after `healthcheck:` if present):
```yaml
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
```

Apply to: web, scraper, embedder, cloudflared, and backup (added by Plan 10-02).

Key decisions:
- `json-file` driver — structured logs queryable by `docker logs`, compatible with Docker's built-in `docker logs` command and any log shipper
- `max-size: 10m` — each log file capped at 10 MB. At typical grocery tracker traffic, this is weeks of logs
- `max-file: 3` — keeps 3 rotated log files (current + 2 historical = max 30 MB per service). Prevents disk exhaustion on long-running deployments
- All 5 services get the same configuration — consistency simplifies operations

For services that already have a `deploy:` block (embedder has `deploy.resources.limits.memory: 512M`), place the `logging:` block BEFORE the `deploy:` block (top-level config, not nested under deploy).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Docker daemon ↔ container health | HEALTHCHECK commands run inside containers. If a container is compromised, health check results could be spoofed. |
| Host disk ↔ Docker logs | JSON log files written to host disk via Docker's logging driver. Disk exhaustion possible without rotation (mitigated by this plan). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-06 | Denial of Service | Host disk (logs) | mitigate | Max 30 MB per service (10 MB × 3 files × 5 services = 150 MB worst-case). Prevents disk exhaustion from unbounded log growth. |
| T-10-07 | Spoofing | Container health check | accept | HEALTHCHECK uses curl to localhost — cannot be spoofed from outside the container. Container compromise could fake responses, but at that point the container is already owned. Accept — focus on preventing container compromise. |
| T-10-08 | Elevation of Privilege | curl binary in containers | mitigate | curl added only to dedicated app containers (not the backup container). Minimal attack surface — curl used only for HEALTHCHECK, not exposed to external input. |
</threat_model>
