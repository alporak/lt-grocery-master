Add a minimal HTTP health server to the scraper process using Node's built-in `http` module (zero new dependencies). The server listens on port 9000 and returns `{ status: "ok" }` on GET /health.

**Why Node http, not Python:** The scraper is a Node.js/TypeScript service (packages/scraper/src/index.ts uses `node-cron`, Playwright). Adding a Python server would require a Python runtime in the Docker image — unnecessarily complex and violates stack consistency.

**Where to inject:** In the `main()` function, BEFORE the cron scheduler setup (so the health endpoint is available immediately when the container starts, before any cron jobs run).

Add this import at the top of `packages/scraper/src/index.ts` (after existing imports):
```typescript
import * as http from "http";
```

Add this block at the start of `main()`, right after `console.log("[Scheduler] Starting scraper service...");`:
```typescript
  // Health check HTTP server — Docker HEALTHCHECK probes port 9000
  const healthServer = http.createServer((_req, res) => {
    if (_req.method === "GET" && (_req.url === "/health" || _req.url === "/")) {
      res.writeHead(200, { "Content-Type": "application/json" });
      res.end(JSON.stringify({ status: "ok" }));
    } else {
      res.writeHead(404);
      res.end();
    }
  });
  healthServer.listen(9000, "0.0.0.0", () => {
    console.log("[Health] Scraper health server listening on :9000");
  });
```

Key decisions:
- Port 9000 per CONTEXT.md agent discretion (dedicated internal port, not exposed in docker-compose)
- Binds to `0.0.0.0` so Docker's HEALTHCHECK can reach it from the Docker network
- Responds to both `/health` and `/` (robustness for different HEALTHCHECK configurations)
- 404 for other paths (minimal attack surface)
- No Prisma/DB access — avoids SQLite writer contention during health checks (Pitfall #8)
</action>
