# Milestone v1.0: Multi-Device Auth

**Status:** ✅ SHIPPED 2026-04-29
**Phases:** 1-4
**Total Plans:** 4 (consolidated)

## Overview

Adds seamless social login (Google/Facebook) to LT Grocery so users can sync their grocery lists across devices. 20 requirements, 4 phases, all shipped.

## Phases

### Phase 1: Auth Foundation

**Goal:** Users can log in with Google/Facebook. Auth state visible in UI.

Plans:
- [x] 1.1: Schema + next-auth setup — Install deps, add User/Account/Session models, configure providers
- [x] 1.2: Auth UI flow — Login page, SessionProvider, auth state in navigation, sign-in/out buttons
- [x] 1.3: Route protection — Middleware.ts, protect grocery-list routes, redirect unauthenticated

**Success criteria:**
1. User clicks "Sign in with Google" → redirected to Google → back to app → authenticated
2. User clicks "Sign in with Facebook" → redirected to Facebook → back to app → authenticated
3. User avatar/name shows in navigation header when logged in
4. Unauthenticated user visiting `/grocery-lists` gets redirect to `/login`
5. Session persists across browser refresh

**Requirements:** AUTH-01 through AUTH-05

### Phase 2: Data Isolation

**Goal:** Each user sees only their own grocery lists and settings. Existing data migrated.

Plans:
- [x] 2.1: Grocery list ownership — Add userId to GroceryList, scope all list APIs, ownership checks
- [x] 2.2: Settings scoping — Split Settings into UserSetting + AppSetting, scope settings API
- [x] 2.3: Data migration — Ghost user, migration script, "Import existing lists?" prompt on first login

**Success criteria:**
1. User A's lists are invisible to User B
2. User A's language/theme/address are independent from User B's
3. System settings (scraper config, pipeline state) still work
4. Existing grocery lists are claimable by first logged-in user
5. Public APIs (search, products, categories, stores, deals) remain public

**Requirements:** DATA-01 through DATA-06

### Phase 3: Cross-Device Sync

**Goal:** Changes to a grocery list on one device appear on another within seconds.

Plans:
- [x] 3.1: Version tracking — Add version field to GroceryList, bump on every mutation
- [x] 3.2: Optimistic concurrency — Server returns 409 on version conflict, client handles with toast + refresh
- [x] 3.3: Sync transport — SSE for tab sync, polling for cross-device, localStorage cache for offline reads

**Success criteria:**
1. Edit list on laptop → changes visible on phone within 5 seconds
2. Simultaneous edits on two devices → second device shown conflict toast, refreshed
3. Same-device tabs sync near-instantly via SSE
4. Offline: last-cached version shown read-only
5. Polling doesn't impact SQLite performance (< 0.6 req/s for 3 devices)

**Requirements:** SYNC-01 through SYNC-05

### Phase 4: Sessions & Polish

**Goal:** User can see and manage active sessions. Edge cases handled.

Plans:
- [x] 4.1: Session management — Device tracking, session list page, sign-out individual sessions
- [x] 4.2: Edge cases — Account linking, error states, loading skeletons, empty states for new users

**Success criteria:**
1. User sees device-active sessions with names in sessions page
2. User can sign out individual sessions
3. Google + Facebook accounts with same email link to one user
4. New users see empty state prompts instead of blank pages
5. Auth errors show user-friendly messages

**Requirements:** SESS-01 through SESS-05

## Milestone Summary

**Key Decisions:**
- next-auth@4 (not v5) — Stable, mature. Proven correct.
- JWT strategy only — Avoids SQLite write contention.
- Google + Facebook only — Covers 95% of users. One-click login.
- Split Settings table — UserSetting (per-user) + AppSetting (global). Clean separation.
- Ghost user migration — Existing lists claimable on first login.
- Polling + SSE hybrid — SSE for tab sync, 5s polling for cross-device.
- Last-Write-Wins conflicts — Optimistic concurrency with version bump.
- Single `NEXTAUTH_URL` env var — One variable to change per deployment.

**Issues Resolved:**
- OAuth redirects now driven by single env var (`NEXTAUTH_URL`) — works on any domain
- Device tracking via `DeviceSession` model with User-Agent parsing
- Conflict resolution for simultaneous edits across devices
- Data isolation between users with ghost migration path

**Issues Deferred:**
- Facebook OAuth credentials not yet provisioned (user is setting up Facebook app)
- No email provider login (by design — OAuth-only for seamlessness)
- No WebSocket real-time sync (polling + SSE sufficient)

**Technical Debt Incurred:**
- Middleware creates a new PrismaClient instance per request (acceptable for self-hosted scale)
- Device session tracking is best-effort (no guarantee of real-time accuracy)
- Offline mode is read-only (write support deferred)
